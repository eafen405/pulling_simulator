## 内容提要

本课题设计并实现了一款命令行抽卡模拟器，模拟热门游戏中基于概率的角色/武器抽取机制。程序采用 C 语言编写，使用模块化架构，将系统划分为状态管理、抽卡算法、卡池数据、记录管理和用户界面五个独立模块。

程序实现了完整的保底系统，包括 5★ 软保底（第 74 抽起概率逐次递增）与硬保底（第 90 抽必出）、4★ 硬保底（每 10 抽必出 4★ 或以上），以及 5★ 大/小保底和 4★ UP 机制。支持单抽与十连两种抽取模式，抽卡数据通过文件持久化保存，可实现断点续抽。

**关键词**：抽卡模拟；保底机制；C 语言；模块化设计；文件持久化

## 1. 需求分析

### 1.1 课题背景

抽卡是当代手机游戏中普遍存在的付费获取角色或道具的机制。其核心是一套基于概率的随机抽取系统，同时辅以"保底"机制保障玩家体验。本课题旨在用 C 语言模拟这一机制，实现一个功能完整、逻辑正确的命令行抽卡模拟器。

### 1.2 功能需求

| 编号 | 功能 | 描述 |
|------|------|------|
| F1 | 单抽 | 执行一次抽卡，返回稀有度与角色/武器名称 |
| F2 | 十连 | 连续执行十次抽卡，按稀有度排序展示结果 |
| F3 | 5★ 软保底 | 距上次 5★ 第 74 抽起概率逐次提升 |
| F4 | 5★ 硬保底 | 第 90 抽必出 5★ |
| F5 | 4★ 硬保底 | 每 10 抽必出 4★ 或以上 |
| F6 | 5★ 大小保底 | 未出 UP 则下一发必出 UP；出了 UP 则重置为小保底 |
| F7 | 4★ UP 机制 | 重云、北斗、迪奥娜，50% 命中率，未命中则下一发大保底 |
| F8 | 状态持久化 | 保底计数保存至 `state.txt`，重启程序不丢失 |
| F9 | 抽卡记录 | 每次抽卡追加写入 `record.txt`，支持历史查看 |
| F10 | 保底状态展示 | 主界面实时显示当前 5★/4★ 垫抽数与大小保底标记 |

### 1.3 运行环境

- 操作系统：Windows 10/11
- 编译器：GCC（MinGW）6.3.0 或以上
- 编译选项：`gcc -Wall -Wextra -I include -o pulls src/*.c`

          ▼               ▼  ▼  ▼               ▼
    ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
    │ 状态模块  │   │ 算法模块  │   │ 记录模块  │   │  UI模块  │
    │ state.c  │   │algorithm│   │ record.c │   │   ui.c   │
    └────┬─────┘   └────┬─────┘   └────┬─────┘   └──────────┘

模块调用规则：
- `main.c` 是唯一调用者，所有模块由 `main` 统一调度
- 唯一例外：**算法模块 → 卡池模块**（只读查询：获取概率配置与角色数量）
- 算法模块不碰文件、不存状态（纯函数）

## 3. 详细设计

### 3.1 数据结构设计

#### 3.1.1 保底状态结构体 `PullState`

```c
struct PullState {
    int pity_5star;         /* 距上次 5★ 已抽次数 (0~89) */
    int pity_4star;         /* 距上次 4★ 已抽次数 (0~9)  */
    int guarantee_5star;    /* 1=大保底(下次必出UP), 0=小保底 */
    int guarantee_4star;    /* 1=4★大保底, 0=4★小保底 */
    int total_pulls;        /* 累计总抽数 */
};
```

字段说明：
- `pity_5star`：距上一次抽出 5★ 后累积的抽数。出 5★ 时重置为 0，每抽一次非 5★ 则 +1
- `pity_4star`：距上一次抽出 4★ 或以上后累积的抽数。出 4★ 或 5★ 时重置为 0
- `guarantee_5star`：上一次 5★ 是否为 UP 角色。如果不是（"歪了"），则此标志置 1，下次 5★ 必出 UP
- `guarantee_4star`：同原理，适用于 4★ UP 角色（重云、北斗、迪奥娜）

#### 3.1.2 卡池概率配置 `PoolRates`

```c
struct PoolRates {
    double rate_5star;       /* 基础 5★ 概率: 0.006 (0.6%) */
    double rate_4star;       /* 4★ 概率: 0.051 (5.1%)      */
    int soft_pity_start;     /* 软保底起始抽数: 74          */
    int hard_pity_5;         /* 5★ 硬保底: 90              */
    int hard_pity_4;         /* 4★ 硬保底: 10              */
};
```

#### 3.1.3 卡池角色数据

卡池数据以静态字符串数组形式硬编码在 `pool.c` 中：

| 稀有度 | 数量 | 内容 |
|--------|------|------|
| 5★ | 9 个 | 桑多涅（UP）+ 刻晴、七七、迪卢克、琴、莫娜、提纳里、迪希雅、梦见月瑞希（常驻） |
| 4★ | 68 个 | 18 件武器 + 50 名角色（含 UP：重云、北斗、迪奥娜） |
| 3★ | 13 个 | 13 件武器 |

### 3.2 核心函数说明

#### 3.2.1 状态管理模块

| 函数 | 功能 | 关键逻辑 |
|------|------|----------|
| `state_load()` | 从 `state.txt` 读取保底状态 | 文件不存在则全部初始化为 0；逐行解析 `key=value` 格式 |
| `state_update()` | 根据抽卡结果更新保底计数 | 5★ 重置两套保底；4★ 重置自身保底、5★ 计数 +1；3★ 两套计数均 +1 |
| `state_save()` | 保存当前状态到 `state.txt` | 写入 5 个字段的 key=value 行 |

#### 3.2.2 抽卡算法模块

| 函数 | 功能 | 关键逻辑 |
|------|------|----------|
| `algorithm_draw()` | 根据当前保底状态判定本次稀有度 | 先判定 4★ 硬保底是否触发；若触发则检查 5★ 概率，命中出 5★ 否则强制 4★；未触发则顺序判定 5★ → 4★ → 3★ |
| `algorithm_pick_character()` | 根据稀有度和保底标志选择具体角色 | 5★：大保底必出 UP(下标0)，小保底 50% UP / 50% 常驻 8 选 1；4★：大保底必出 3 个 UP 之一，小保底 50% UP，未中则从其余 65 个中随机；3★：13 选 1 |

#### 3.2.3 卡池数据模块

| 函数 | 功能 |
|------|------|
| `pool_get_rates()` | 返回概率配置结构体指针 |
| `pool_get_name(rarity, index)` | 根据稀有度和池内下标返回角色/武器名称 |
| `pool_get_count(rarity)` | 返回某稀有度的物品总数 |

#### 3.2.4 用户界面模块

| 函数 | 功能 |
|------|------|
| `ui_menu(state)` | 显示主菜单及保底状态，接收用户输入 |
| `ui_show_result(rarity, name)` | 显示单抽结果（5★ 金色、4★ 紫色、3★ 默认色） |
| `ui_show_ten_result(r, names, 10)` | 按 5★ → 4★ → 3★ 排序展示十连结果 |

### 3.3 关键算法

#### 3.3.1 5★ 软保底概率计算

```c
static double calc_rate_5(int pity_5star, const struct PoolRates* rates) {
    if (pity_5star >= rates->hard_pity_5 - 1)       // 第 90 抽
        return 1.0;                                  // 必出

    if (pity_5star >= rates->soft_pity_start - 1) {  // 第 74 抽起
        int pulls_into_soft = pity_5star - (rates->soft_pity_start - 1) + 1;
        return rates->rate_5star * (1.0 + 10.0 * pulls_into_soft);
        // 基础概率 0.6%，每多 1 抽 +6%：74 抽 = 6.6%, 75 = 12.6%...
    }

    return rates->rate_5star;  // 前 73 抽：恒定 0.6%
}
```

#### 3.3.2 4★ 保底判定与优先级

在 4★ 硬保底触发时（距上次 4★ 已累计 9 抽），程序**先判定 5★**：
- 如果 5★ 命中 → 出 5★（5★ 等级高于 4★，且同时重置两套保底）
- 如果 5★ 未命中 → 强制出 4★（保证至少 4★）

#### 3.3.3 随机判定实现

```c
static int roll_percent(double probability) {
    int threshold = (int)(probability * 10000);  // 0.006 → 60
    return (rand() % 10000) < threshold;         // 万分之六十
}
```

使用 `rand() % 10000` 生成 0~9999 的均匀分布随机数，与万分之一精度的阈值比较。

#### 3.3.4 4★ UP 非 UP 池索引映射

4★ 共 68 个物品（下标 0~67），其中 UP 角色 3 个（下标 53=迪奥娜、58=北斗、67=重云）。未中 UP 时需从剩余 65 个中随机，算法采用"压缩范围 + 跳过 UP 下标"的方式：

```c
int idx = rand() % (68 - 3);     // 0~64
for (int i = 0; i < 3; i++) {
    if (idx >= up4_idx[i]) idx++;  // 跳过 UP 下标
}
```

## 5. 总结与心得

### 技术收获

1. **模块化编程**：将系统拆分为 5 个独立模块，每个模块职责单一、接口清晰。`main.c` 只做调度，不涉及具体逻辑，体现了"高内聚、低耦合"的设计原则。

2. **结构体的实际运用**：`PullState` 和 `PoolRates` 结构体的设计让我深刻理解了"数据 + 操作"的编程范式。结构体作为参数在模块间传递，保证了数据的完整性和一致性。

3. **文件 I/O 实践**：通过 `state.txt` 和 `record.txt` 的读写，掌握了 `fopen/fclose/fgets/fprintf` 等标准库函数的使用，理解了文本格式设计（`key=value`）对解析便利性的影响。

4. **指针的深入理解**：函数参数传递 `const struct PullState*` 而非值拷贝，既保证了效率，又通过 `const` 修饰保护了只读语义。

5. **概率编程**：实现了现实游戏中的软保底、硬保底、大小保底等多层判定逻辑，理解了"先判定高稀有度"的优先级设计。

### 不足与改进方向

- 卡池数据目前硬编码在源码中，未来可改为从 `pool.json` 读取，实现热更新
- 可使用时间戳作为随机数种子的一部分，增加随机性
- 可改为彩色 ANSI 清屏替代 `system("cls")`，提高可移植性
